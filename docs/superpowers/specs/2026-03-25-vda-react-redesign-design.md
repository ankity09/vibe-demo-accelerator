# VDA React Redesign — Lovable-Inspired Developer Experience

**Date:** 2026-03-25
**Status:** Approved
**Author:** Ankit Yadav + Claude Code

## Problem Statement

The Vibe Demo Accelerator (VDA) has three pain points that limit its effectiveness for SAs building customer demos:

1. **UI quality** — The single-HTML frontend with embedded CSS/JS limits design quality. Too many iteration rounds to get something demo-worthy.
2. **Slow iteration loop** — Edit in vibe, deploy to Databricks, check in browser, describe changes, redeploy. No live feedback.
3. **Scaffold rigidity** — Hard to rearrange layouts, add new pages touches 6 places, and only 4 Databricks features (Genie, MAS, KA, UC Functions) are offered during setup. Other capabilities require manual integration.

## Target Users

Any SA in the Databricks org. The system must work for someone who has never seen the codebase, clones it, and runs `/new-demo`. Zero prior context required.

## Design Decisions

| Decision | Choice |
|---|---|
| Frontend architecture | React 18 + Vite + TypeScript (full migration from single HTML) |
| Dev loop | Companion browser: Vite HMR + Chrome DevTools MCP for vibe self-verification |
| Design system | shadcn/ui primitives + 12 VDA compound components, light/dark mode |
| Layouts | Composable AppShell with 3 variants (sidebar, top-nav, dashboard-first) |
| Databricks features | Dynamic menu powered by AI Dev Kit detection (14+ features) |
| Wizard | Terminal-based, 3-phase, smart defaults + use-case recommendations |
| Scope | Big bang redesign — current working version on GitHub as safety net |
| Backend | Core modules untouched, main.py becomes assembly, routes extracted to files |

## Section 2: Component Library

Two tiers: shadcn/ui primitives (standard) and 12 VDA compound components (pre-built for demo patterns).

### VDA Compound Components (`components/vda/`)

#### 1. `<AgentChat />`
Full MAS streaming chat with action cards, reasoning blocks, suggested follow-ups.
```tsx
interface AgentChatProps {
  endpoint?: string                          // Default: "/api/chat"
  welcomeTitle?: string
  welcomeSubtitle?: string
  suggestedPrompts?: string[]
  agentNameMap?: Record<string, string>      // MAS tool name → display label
  actionCardTables?: ActionCardConfig[]      // Entity cards from Lakebase
  showReasoning?: boolean                    // Default: true
  autoApproveMcp?: boolean                   // Default: true
}
```
Handles: SSE streaming (`thinking`/`delta`/`tool_call`/`action_card` events), chat history, session expiry detection, suggested follow-ups. Ported from current implementation using `useSSE` and `useChatStore` hooks.

#### 2. `<KPIDashboard />`
Grid of metric cards with animated staggered entrance.
```tsx
interface KPIDashboardProps {
  metrics: KPIMetric[]     // { label, value, change, trend, icon, color }
  columns?: 3 | 4 | 5
  loading?: boolean        // Skeleton shimmer
}
```

#### 3. `<WorkflowPanel />`
Workflow cards with severity/status filters and detail modal.
```tsx
interface WorkflowPanelProps {
  endpoint?: string                          // Default: "/api/agent-overview"
  workflowTypes?: string[]
  severityLevels?: string[]
  agentSteps?: Record<string, string[]>      // Workflow type → agent sequence
  onAskAI?: (prompt: string) => void         // Bridge to chat
}
```

#### 4. `<DataExplorer />`
Paginated, sortable, filterable data table built on shadcn `<Table>`.
```tsx
interface DataExplorerProps {
  endpoint: string
  columns: ColumnDef[]     // { label, key, type: "text"|"number"|"date"|"badge"|"link", sortable, filterable }
  searchable?: boolean
  pageSize?: number        // Default: 25
  onRowClick?: (row: any) => void
  actions?: RowAction[]
}
```

#### 5. `<ExceptionManager />`
Alert/exception triage interface with inline actions and AI bridge.
```tsx
interface ExceptionManagerProps {
  endpoint?: string
  severityColors?: Record<string, string>
  actions?: ("acknowledge" | "escalate" | "resolve" | "dismiss")[]
  onAskAI?: (prompt: string) => void
}
```

#### 6. `<MapContainer />`
Geospatial visualization wrapping Mapbox GL or Leaflet. Handles tab-switch invalidation.
```tsx
interface MapContainerProps {
  center?: [number, number]
  zoom?: number
  markers?: MapMarker[]
  routes?: MapRoute[]       // { points, color, animated }
  heatmap?: HeatmapData
  onMarkerClick?: (marker: MapMarker) => void
}
```

#### 7. `<TimelineView />`
Chronological event timeline for audit trails, agent actions, incident history.
```tsx
interface TimelineViewProps {
  events: TimelineEvent[]   // { timestamp, title, description, type, severity }
  groupBy?: "day" | "hour"
  maxItems?: number
  onEventClick?: (event: TimelineEvent) => void
}
```

#### 8. `<MetricsBar />`
Compact horizontal metrics strip for page headers.
```tsx
interface MetricsBarProps {
  metrics: { label: string; value: string | number; icon?: string }[]
  variant?: "compact" | "full"
}
```

#### 9. `<ThemeToggle />`
Light/dark mode switch with optional theme preset dropdown.
```tsx
interface ThemeToggleProps {
  position?: "nav" | "floating"
}
```

#### 10. `<NotesPanel />`
Slide-out panel for entity-attached notes, reads/writes to Lakebase `notes` table.
```tsx
interface NotesPanelProps {
  entityType: string
  entityId: string
  endpoint?: string         // Default: "/api/notes"
}
```

#### 11. `<LiveFeedIndicator />`
Streaming status with pulsing indicator, optional start/stop controls and stats.
```tsx
interface LiveFeedIndicatorProps {
  showControls?: boolean
  showStats?: boolean
  pollInterval?: number     // Default: 15000ms
}
```

#### 12. `<ActionCard />`
Entity card rendered in chat when MAS creates/references entities. Used internally by `<AgentChat />` but also available standalone.
```tsx
interface ActionCardProps {
  card: ActionCardData      // { type, id, title, details, actions }
  onAction: (action: string, id: string) => void
}
```

### Usage Pattern

Vibe composes pages from VDA components for common patterns and drops to shadcn primitives for custom needs:
```tsx
import { KPIDashboard } from '@/components/vda/KPIDashboard'
import { DataExplorer } from '@/components/vda/DataExplorer'
import { useApi } from '@/hooks/useApi'

export function Dashboard() {
  const { data: metrics, loading } = useApi('/api/dashboard/metrics')
  return (
    <div className="space-y-6">
      <KPIDashboard metrics={metrics?.kpis ?? []} loading={loading} />
      <DataExplorer
        endpoint="/api/assets"
        columns={[
          { key: 'name', label: 'Asset', sortable: true },
          { key: 'status', label: 'Status', type: 'badge' },
          { key: 'health_score', label: 'Health', type: 'number' },
        ]}
      />
    </div>
  )
}
```

### Tailwind Integration

`tailwind.config.ts` maps CSS variables to utility classes:

```ts
export default {
  theme: {
    extend: {
      colors: {
        bg: {
          primary: 'var(--bg-primary)',
          secondary: 'var(--bg-secondary)',
          card: 'var(--bg-card)',
          hover: 'var(--bg-hover)',
        },
        text: {
          primary: 'var(--text-primary)',
          secondary: 'var(--text-secondary)',
          muted: 'var(--text-muted)',
        },
        accent: {
          DEFAULT: 'var(--accent)',
          hover: 'var(--accent-hover)',
          muted: 'var(--accent-muted)',
        },
      }
    }
  }
}
```

Components use semantic tokens (`bg-bg-card`, `text-text-primary`, `border-border`) — never raw hex values. Every component works in every theme and both modes automatically.

## Section 4: Layout System

`<AppShell>` is the outermost wrapper. It reads layout preference from config and renders the right structure. Adding a page means adding one route entry.

### AppShell

```tsx
interface AppShellProps {
  layout: "sidebar" | "topnav" | "dashboard"
  logo?: React.ReactNode
  demoName: string
  children: React.ReactNode
}
```

**Variants:**
- **Sidebar** — Fixed left sidebar (collapsible), top header bar. Best for data-heavy demos.
- **TopNav** — Horizontal nav bar, full-width content. Best for fewer pages.
- **Dashboard** — No persistent nav, floating hamburger menu. Maximum data density.

### Route-Based Navigation

Pages register through a config array — one place, not six:

```tsx
const routes: RouteConfig[] = [
  { path: '/',          label: 'Dashboard',  icon: LayoutDashboard, page: lazy(() => import('./pages/Dashboard')) },
  { path: '/chat',      label: 'AI Chat',    icon: MessageSquare,   page: lazy(() => import('./pages/Chat')) },
  { path: '/workflows', label: 'Workflows',  icon: GitBranch,       page: lazy(() => import('./pages/Workflows')) },
  // Vibe adds domain pages here:
  { path: '/assets',    label: 'Assets',     icon: Wrench,          page: lazy(() => import('./pages/Assets')) },
]
```

AppShell reads this array and renders nav links automatically — sidebar items, top-nav links, or drawer menu items depending on layout. Lazy loading code-splits each page.

### Adding a New Page

1. Create `pages/MyPage.tsx`
2. Add one entry to the routes array
3. Done — nav updates automatically

### Architecture

```
┌─────────────────┐     ┌──────────────────┐     ┌─────────────────────┐
│   SA in vibe     │     │   Vite Dev Server │     │   Chrome Browser     │
│   (terminal)     │     │   (background)    │     │   (companion)        │
│                  │     │                   │     │                      │
│  "add an assets  │────▶│  vibe writes      │────▶│  HMR hot-swaps the   │
│   page with a    │     │  pages/Assets.tsx  │     │  component in <1s    │
│   health table"  │     │  + route entry     │     │  — no full reload    │
│                  │     │                   │     │                      │
│                  │     │  Proxy /api/* ────▶│  FastAPI (local or       │
│                  │◀────│                   │◀────│  remote)              │
│  vibe takes a    │     │                   │     │  Chrome DevTools MCP  │
│  screenshot via  │     │                   │     │  exposes DOM, console │
│  DevTools MCP    │     │                   │     │  errors, screenshots  │
└─────────────────┘     └──────────────────┘     └─────────────────────┘
```

### Startup Sequence

When vibe begins generating frontend code:
1. `cd app/frontend && npm install` (if needed)
2. Start Vite dev server in background: `npm run dev -- --port 5173`
3. Start FastAPI in background: `uvicorn backend.main:app --reload --port 8000`
4. Open Chrome to `http://localhost:5173`
5. Connect Chrome DevTools MCP for visual verification

Steps 2-4 happen once per session. HMR keeps the browser in sync after that.

### Vite Proxy Config

```ts
// vite.config.ts
export default defineConfig({
  server: {
    port: 5173,
    proxy: {
      '/api': {
        target: 'http://localhost:8000',
        changeOrigin: true,
      }
    }
  }
})
```

### Vibe Self-Verification

After generating or modifying a component, vibe uses `web-devloop-tester` agent to:
1. Screenshot the current page
2. Check browser console for errors
3. Test navigation to the new page
4. Self-correct if layout is broken or console has errors

### API Backend Options During Dev

- **Option A (default): Local FastAPI** — Both servers running locally. Requires Databricks CLI profile for SDK auth.
- **Option B: Proxy to deployed app** — Change Vite proxy target to deployed Databricks App URL. Frontend-only dev.

```

─────────                           ──────────────────
CREATE Databricks services:         WIRE INTO the running app:
```
